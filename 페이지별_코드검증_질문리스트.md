# 페이지별 코드 검증 질문 리스트

> AI로 생성한 웹/앱 프로젝트를 페이지 단위로 검증할 때 사용하는 체크리스트입니다.
> 한 번에 하나의 카테고리씩 실행하는 것을 권장합니다.
> 권장 순서: 구조 → 보안 → API → DB → 에러 처리 → 성능 → 반응형/접근성 → 앱 연동 → AWS

---

## 프론트엔드 검증

### 1. 구조 & 코드 품질

- [ ] 이 페이지의 컴포넌트 구조가 재사용 가능하게 분리되어 있는가? 불필요하게 중복된 코드는 없는가?
- [ ] 상태 관리(state management)가 적절한가? props drilling이 과도하지 않은가?
- [ ] 하드코딩된 값이 있는가? 환경변수나 config로 분리해야 할 것은 없는가?

**프롬프트 예시:**
```
이 페이지의 컴포넌트 구조를 시니어 프론트엔드 개발자 관점에서 리뷰해줘.
재사용성, 상태 관리, 하드코딩 여부를 중심으로 문제를 짚고 수정 코드를 제시해줘.
```

---

### 2. 반응형 & 접근성

- [ ] 모바일(375px), 태블릿(768px), 데스크톱(1280px) 각 해상도에서 레이아웃이 깨지지 않는가?
- [ ] 키보드만으로 모든 기능을 사용할 수 있는가? Tab 순서가 논리적인가?
- [ ] 이미지에 alt 텍스트가 있는가? 색상 대비가 WCAG 기준을 충족하는가?

**프롬프트 예시:**
```
이 페이지를 375px, 768px, 1280px 해상도 기준으로 반응형 검토해줘.
접근성(키보드 탐색, alt 텍스트, 색상 대비)도 함께 점검하고 수정 코드를 제시해줘.
```

---

### 3. 프론트엔드 성능

- [ ] 불필요한 리렌더링이 발생하는 컴포넌트가 있는가?
- [ ] 이미지가 최적화되어 있는가? (WebP, lazy loading, 적절한 사이즈)
- [ ] 번들 사이즈에 불필요하게 큰 라이브러리가 포함되어 있지 않은가?

**프롬프트 예시:**
```
이 페이지의 프론트엔드 성능을 분석해줘.
리렌더링, 이미지 최적화, 번들 사이즈 관점에서 개선점을 찾고 수정 코드를 제시해줘.
```

---

### 4. 에러 처리 & UX

- [ ] API 호출 실패 시 사용자에게 어떤 피드백을 주는가? 로딩 상태는 표시되는가?
- [ ] 빈 상태(데이터가 없을 때), 에러 상태, 로딩 상태가 모두 처리되어 있는가?
- [ ] 폼이 있다면 유효성 검사(validation)가 클라이언트와 서버 양쪽에서 이루어지는가?

**프롬프트 예시:**
```
이 페이지의 에러 처리와 UX 상태 관리를 점검해줘.
로딩/에러/빈 상태 처리, 폼 유효성 검사가 빠진 곳을 찾고 수정 코드를 제시해줘.
```

## 백엔드 검증

### 5. API 설계

- [ ] API 엔드포인트 네이밍이 RESTful 규칙을 따르는가? 일관성이 있는가?
- [ ] 각 API의 요청/응답 형식이 문서화되어 있는가?
- [ ] 페이지네이션, 필터링, 정렬이 필요한 API에 구현되어 있는가?

**프롬프트 예시:**
```
이 페이지에서 사용하는 API 엔드포인트를 모두 찾아줘.
RESTful 규칙 준수 여부, 요청/응답 형식, 페이지네이션 구현 여부를 점검하고 개선안을 제시해줘.
```

---

### 6. 보안

- [ ] 인증(Authentication)과 인가(Authorization)가 모든 보호된 엔드포인트에 적용되어 있는가?
- [ ] SQL Injection, XSS, CSRF 방어가 되어 있는가?
- [ ] API 키, DB 비밀번호 등 민감 정보가 코드에 하드코딩되어 있지 않은가? (.env 또는 AWS Secrets Manager를 사용하는가?)
- [ ] CORS 설정이 필요한 도메인만 허용하는가? 와일드카드(*)로 열려있지 않은가?

**프롬프트 예시:**
```
이 페이지 관련 코드의 보안을 전면 점검해줘.
인증/인가, SQL Injection, XSS, CSRF, 민감 정보 노출, CORS 설정을 모두 확인하고
취약점이 있으면 심각도와 함께 수정 코드를 제시해줘.
```

---

### 7. 데이터베이스

- [ ] 테이블 간 관계(FK)가 올바르게 설정되어 있는가?
- [ ] 자주 조회되는 컬럼에 인덱스가 걸려 있는가?
- [ ] 사용자 데이터 삭제 시 관련 데이터가 함께 처리되는가? (cascade 또는 soft delete)
- [ ] 이 페이지에서 다루는 사용자 데이터(수면 기록, 식습관 기록, 셀프체크 결과 등)의 백업/복구 전략이 설계되어 있는가?
- [ ] 데이터 손실 시 복구 가능한 구조인가? (Point-in-Time Recovery, 버전 관리 등)

**프롬프트 예시:**
```
이 페이지에서 사용하는 DB 테이블과 쿼리를 분석해줘.
FK 관계, 인덱스 설정, 삭제 전략(cascade/soft delete)을 점검하고,
이 페이지에서 다루는 사용자 데이터의 백업/복구 전략이 적절한지도 함께 검토해줘.
개선안을 제시해줘.
```

---

### 8. 에러 & 로깅

- [ ] 예상치 못한 에러 발생 시 서버가 크래시되지 않고 적절한 에러 응답을 반환하는가?
- [ ] 로그가 남고 있는가? AWS CloudWatch 등으로 모니터링할 수 있는가?
- [ ] 에러 응답에 내부 구현 정보(스택 트레이스, DB 스키마 등)가 노출되지 않는가?

**프롬프트 예시:**
```
이 페이지 관련 백엔드 코드의 에러 처리와 로깅을 점검해줘.
try-catch 누락, 에러 응답 형식, 민감 정보 노출, CloudWatch 연동 여부를 확인하고 수정 코드를 제시해줘.
```

## 앱 연동 대비 검증

### 9. 모바일 앱 확장성

- [ ] API가 웹과 모바일 앱 모두에서 사용할 수 있도록 설계되어 있는가? (웹 전용 로직이 API에 섞여 있지 않은가?)
- [ ] 인증 방식이 모바일 앱에서도 작동하는가? (JWT 기반인가, 세션 기반인가?)
- [ ] 파일 업로드가 있다면 S3 등 클라우드 스토리지를 사용하는가? 서버 로컬에 저장하고 있지 않은가?
- [ ] 푸시 알림, 딥링킹 등 모바일 기능을 위한 확장 여지가 있는가?

**프롬프트 예시:**
```
이 페이지의 코드를 모바일 앱 연동 관점에서 검토해줘.
API 범용성, 인증 방식(JWT vs 세션), 파일 저장 방식, 모바일 확장성을 점검하고
앱 연동 시 문제가 될 부분과 수정안을 제시해줘.
```

## AWS 인프라 검증

### 10. 배포 & 운영

- [ ] 개발(dev), 스테이징(staging), 프로덕션(prod) 환경이 분리되어 있는가?
- [ ] HTTPS가 적용되어 있는가? SSL 인증서가 설정되어 있는가?
- [ ] 자동 스케일링이 설정되어 있는가? 트래픽 증가 시 대응 가능한가?
- [ ] 백업 전략이 있는가? DB 스냅샷이 자동으로 생성되는가?
- [ ] DynamoDB 테이블별 Point-in-Time Recovery(PITR)가 활성화되어 있는가?
- [ ] AWS Backup을 통한 자동 백업 스케줄과 보존 기간이 적절한가?
- [ ] 백업 데이터로부터 실제 복구 테스트를 해본 적이 있는가?
- [ ] 비용 알림(AWS Budget Alert)이 설정되어 있는가?

**프롬프트 예시:**
```
현재 AWS 배포 구성을 점검해줘.
환경 분리, HTTPS/SSL, 자동 스케일링, 비용 알림 설정 여부를 확인하고,
DynamoDB 테이블별 PITR 활성화 여부, AWS Backup 스케줄과 보존 기간,
백업 복구 테스트 가능 여부를 점검해줘.
누락된 항목의 설정 방법을 단계별로 알려줘.
```
