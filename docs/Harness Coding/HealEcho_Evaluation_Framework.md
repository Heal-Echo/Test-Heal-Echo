# Heal Echo — Evaluation Framework (품질 평가 프레임워크)

> **Version:** 1.0
> **Date:** 2026-04-04
> **Purpose:** AI Agent(Claude Code, Cowork)가 Heal Echo 코드의 기능 동작, 보안, 성능, 접근성 등 품질 전반을 평가할 때 사용하는 프레임워크
> **Usage:** 구조적 수렴 검증(Structural Convergence Rules) 완료 후, 이 문서를 컨텍스트로 제공하여 품질 평가를 수행합니다.
> **선행 조건:** `HealEcho_Structural_Convergence_Rules.md`의 모든 Phase가 100% PASS 상태여야 합니다.

---

## 0. 평가 등급 체계

| 등급 | 점수 | 의미 |
|------|------|------|
| **A (Excellent)** | 90-100 | 프로덕션 배포 가능. 앱 연동 준비 완료 |
| **B (Good)** | 75-89 | 소규모 수정 후 배포 가능 |
| **C (Needs Work)** | 60-74 | 중요 이슈 존재. 수정 필수 |
| **D (Critical)** | 0-59 | 심각한 문제. 대규모 수정 필요 |

각 카테고리를 100점 만점으로 평가하고, 가중 평균으로 종합 점수를 산출합니다.

## 2. E1 — Security (보안) [25%]

### 평가 항목

#### E1-01: 인증 흐름 무결성
- **검증 대상:** `/src/auth/`, OAuth callback routes, `tokenManager.ts`
- **체크리스트:**
  - [ ] OAuth state 파라미터를 사용하여 CSRF 공격을 방지하는가?
  - [ ] Authorization Code → Token 교환이 서버 사이드(API Route)에서 이루어지는가?
  - [ ] Client Secret이 클라이언트 코드에 노출되지 않는가?
  - [ ] 토큰 만료 시 자동 갱신(refresh) 로직이 존재하는가?
  - [ ] 로그아웃 시 모든 토큰과 사용자 데이터가 정리되는가?

#### E1-02: API 엔드포인트 보호
- **검증 대상:** `/src/app/api/user/`, `/src/app/api/admin/`
- **체크리스트:**
  - [ ] 모든 /user/* 엔드포인트가 JWT Bearer 토큰을 검증하는가?
  - [ ] 모든 /admin/* 엔드포인트가 admin 그룹 멤버십을 확인하는가?
  - [ ] /public/* 엔드포인트에 민감한 데이터가 노출되지 않는가?
  - [ ] API Route에서 입력 본문 크기를 제한하는가?
  - [ ] Content-Type 검증이 이루어지는가?

#### E1-03: 클라이언트 사이드 보안
- **검증 대상:** 전체 프론트엔드 코드
- **체크리스트:**
  - [ ] `dangerouslySetInnerHTML` 사용 시 XSS 방지가 되어 있는가?
  - [ ] 사용자 입력이 직접 DOM에 삽입되지 않는가?
  - [ ] NEXT_PUBLIC_ 접두사가 붙은 환경변수에 비밀키가 포함되지 않았는가?
  - [ ] 민감한 정보(카드번호, 개인정보)가 localStorage에 평문으로 저장되지 않는가?

#### E1-04: 결제 보안
- **검증 대상:** Toss Payments 연동 코드, billing 관련 API
- **체크리스트:**
  - [ ] 결제 금액 검증이 서버 사이드(Lambda)에서 이루어지는가?
  - [ ] 빌링키가 클라이언트에 직접 노출되지 않는가?
  - [ ] 결제 상태 변경이 원자적(atomic)으로 처리되는가?
  - [ ] 중복 결제 방지 로직이 존재하는가?

#### E1-05: Admin 분리
- **검증 대상:** Admin Cognito Pool, middleware.ts
- **체크리스트:**
  - [ ] Admin과 일반 사용자가 별도의 Cognito Pool을 사용하는가?
  - [ ] Admin 토큰이 일반 사용자 엔드포인트에 접근할 수 없는가?
  - [ ] Admin 페이지에 no-cache 헤더가 설정되어 있는가?

### 채점 기준
- **100점:** 모든 체크리스트 PASS
- **-10점:** 각 FAIL 항목당 (보안 취약점은 가중 감점)
- **-25점:** Client Secret 노출, 인증 우회 가능 등 Critical 취약점

## 3. E2 — Functionality (기능 정합성) [20%]

### 평가 항목

#### E2-01: 구독 라이프사이클 정합성
- **검증 대상:** Subscription 관련 전체 코드
- **체크리스트:**
  - [ ] 6가지 구독 상태(browser, browser_selected, free_trial, paid, paid_stopped, free_stopped) 전이가 정확한가?
  - [ ] 각 상태에서 접근 가능한 콘텐츠가 올바르게 제한되는가?
  - [ ] 무료 체험 7일 카운트다운이 정확하게 계산되는가?
  - [ ] 구독 취소 후에도 잔여 기간 동안 콘텐츠 접근이 가능한가?
  - [ ] 결제 실패 시 상태 전이가 올바르게 처리되는가?

#### E2-02: 콘텐츠 접근 제어
- **검증 대상:** 비디오 플레이어, 프로그램 선택 로직
- **체크리스트:**
  - [ ] 비로그인 사용자가 유료 콘텐츠에 접근할 수 없는가?
  - [ ] 프로그램 미선택 상태에서 프로그램 전용 콘텐츠가 차단되는가?
  - [ ] 주차별 콘텐츠가 순서대로 해금되는 로직이 정확한가?
  - [ ] 보상 사이클(GiftCycle) 해금·만료가 정확하게 동작하는가?

#### E2-03: 온보딩 흐름
- **검증 대상:** Profile Setup, Program Selection
- **체크리스트:**
  - [ ] 프로필 설정과 프로그램 선택이 독립적 상태 머신으로 분리되어 있는가?
  - [ ] 온보딩 미완료 사용자가 메인 콘텐츠에 접근할 수 없는가?
  - [ ] 프로그램 1회 변경 제한 로직이 정확한가?
  - [ ] 다른 기기에서 로그인 시 AWS에서 프로그램 선택 상태가 복원되는가?

#### E2-04: 비디오 관리 (Admin)
- **검증 대상:** Admin Studio 관련 코드
- **체크리스트:**
  - [ ] 영상 업로드(S3 multipart) 흐름이 완전한가?
  - [ ] 영상 메타데이터 CRUD가 모두 동작하는가?
  - [ ] 프로그램별·주차별 영상 분류가 정확한가?
  - [ ] 썸네일 자동 생성 또는 수동 업로드가 동작하는가?

#### E2-05: 자가 진단 및 추적
- **검증 대상:** PSQI, SelfCheck, SleepLog, HabitTracking
- **체크리스트:**
  - [ ] PSQI 설문 점수 계산이 정확한가?
  - [ ] 자가진단 결과가 올바르게 저장되고 조회되는가?
  - [ ] 수면 기록이 날짜별로 정확하게 저장되는가?
  - [ ] 습관 추적 데이터가 주차별로 올바르게 집계되는가?

### 채점 기준
- **100점:** 모든 비즈니스 시나리오가 정확하게 동작
- **-5점:** 일반 기능 오류
- **-15점:** 결제, 구독 상태 등 핵심 기능 오류
- **-20점:** 데이터 손실 가능성이 있는 기능 오류

## 4. E3 — Error Handling (에러 처리) [15%]

### 평가 항목

#### E3-01: API 에러 처리
- **검증 대상:** 모든 API Route 및 Lambda 함수
- **체크리스트:**
  - [ ] try-catch로 예상치 못한 에러를 포착하는가?
  - [ ] 적절한 HTTP 상태 코드를 반환하는가? (400/401/403/404/500)
  - [ ] 에러 응답 형식이 통일되어 있는가? `{ error, message }`
  - [ ] Lambda 함수에서 발생한 에러가 API Route까지 올바르게 전파되는가?
  - [ ] 업스트림 서비스(Lambda) 타임아웃 시 클라이언트에 적절한 응답을 주는가?

#### E3-02: 클라이언트 에러 처리
- **검증 대상:** 프론트엔드 전체
- **체크리스트:**
  - [ ] API 호출 실패 시 사용자에게 에러 메시지를 표시하는가?
  - [ ] 네트워크 오류(오프라인) 시 적절한 대응이 있는가?
  - [ ] 로딩 상태가 모든 비동기 작업에 표시되는가?
  - [ ] 빈 상태(데이터 없음)에 대한 UI가 존재하는가?
  - [ ] 토큰 만료 시 자동 로그아웃 또는 갱신이 이루어지는가?

#### E3-03: 결제 에러 처리
- **검증 대상:** Billing 관련 코드
- **체크리스트:**
  - [ ] 결제 실패 시 사용자에게 명확한 안내를 제공하는가?
  - [ ] 카드 등록 실패 시 재시도 흐름이 존재하는가?
  - [ ] 정기결제 실패 시 구독 상태 전이가 올바른가?
  - [ ] 결제 콜백 중 네트워크 오류 시 데이터 정합성이 유지되는가?

### 채점 기준
- **100점:** 모든 에러 시나리오에 적절한 처리 존재
- **-5점:** 에러 메시지 누락 또는 불명확
- **-10점:** try-catch 없는 비동기 작업
- **-20점:** 에러 발생 시 앱 크래시 또는 데이터 손실

## 5. E4 — Performance (성능) [15%]

### 평가 항목

#### E4-01: 초기 로딩 성능
- **체크리스트:**
  - [ ] 불필요한 라이브러리가 번들에 포함되지 않았는가?
  - [ ] 이미지가 Next.js Image 컴포넌트 또는 lazy loading을 사용하는가?
  - [ ] 코드 스플리팅(dynamic import)이 적용되어 있는가?
  - [ ] 사용하지 않는 import가 없는가?

#### E4-02: 데이터 페칭 최적화
- **체크리스트:**
  - [ ] 동일한 데이터를 여러 번 중복 요청하지 않는가?
  - [ ] 적절한 캐싱 전략이 적용되어 있는가?
  - [ ] 대량 데이터 조회 시 페이지네이션이 적용되어 있는가?
  - [ ] 검색/필터에 debounce가 적용되어 있는가?

#### E4-03: 렌더링 최적화
- **체크리스트:**
  - [ ] 불필요한 리렌더링이 발생하지 않는가? (React.memo, useMemo, useCallback 적절 사용)
  - [ ] 큰 리스트에 가상 스크롤링(virtualization)이 필요하지 않은가?
  - [ ] useEffect의 의존성 배열이 정확한가?
  - [ ] 상태 관리가 최소 범위로 한정되어 있는가? (전역 상태 최소화)

#### E4-04: Lambda 성능
- **체크리스트:**
  - [ ] Lambda 콜드 스타트를 최소화하는 설정이 있는가?
  - [ ] DynamoDB 쿼리에 적절한 인덱스를 사용하는가?
  - [ ] Lambda 함수의 메모리 설정이 적절한가?
  - [ ] 불필요한 라이브러리 import로 번들 크기가 비대하지 않은가?

### 채점 기준
- **100점:** 초기 로드 2초 이내, 불필요한 요청 없음
- **-5점:** 미사용 import, 중복 요청 등 경미한 이슈
- **-15점:** 심각한 성능 저하 (대량 데이터 무한 로딩, 캐싱 부재 등)

## 6. E5 — Responsive & Accessibility (반응형·접근성) [10%]

### 평가 항목

#### E5-01: 반응형 디자인
- **체크리스트:**
  - [ ] 모바일(360px) ~ 데스크톱(1440px) 브레이크포인트가 적용되어 있는가?
  - [ ] Tailwind CSS 반응형 클래스(sm:, md:, lg:)가 적절히 사용되는가?
  - [ ] 터치 타겟이 최소 44x44px인가?
  - [ ] 가로 스크롤이 발생하지 않는가?

#### E5-02: 웹 접근성
- **체크리스트:**
  - [ ] 이미지에 alt 텍스트가 있는가?
  - [ ] 폼 요소에 label이 연결되어 있는가?
  - [ ] 적절한 HTML 시맨틱 태그(header, nav, main, section)가 사용되는가?
  - [ ] 색상 대비가 WCAG AA 기준(4.5:1)을 충족하는가?
  - [ ] 키보드 네비게이션이 가능한가?

### 채점 기준
- **100점:** 모든 디바이스에서 정상 작동, WCAG AA 충족
- **-10점:** 특정 화면 크기에서 레이아웃 깨짐
- **-15점:** 주요 기능이 모바일에서 사용 불가

## 7. E6 — Database Design (데이터베이스) [10%]

### 평가 항목

#### E6-01: DynamoDB 테이블 설계
- **체크리스트:**
  - [ ] 각 테이블의 파티션 키/정렬 키가 쿼리 패턴에 최적화되어 있는가?
  - [ ] GSI(Global Secondary Index)가 필요한 쿼리 패턴에 적절히 생성되어 있는가?
  - [ ] 핫 파티션이 발생하지 않는 키 설계인가?
  - [ ] TTL(Time To Live) 설정이 필요한 데이터(세션, 임시 데이터)에 적용되어 있는가?

#### E6-02: 데이터 정합성
- **체크리스트:**
  - [ ] 구독 상태 변경 시 관련 테이블이 일관되게 업데이트되는가?
  - [ ] 시청 기록이 중복 저장되지 않는가?
  - [ ] 결제 트랜잭션 로그가 누락 없이 저장되는가?
  - [ ] 사용자 탈퇴 시 관련 데이터가 정리되는가?

#### E6-03: 백업 및 복구
- **체크리스트:**
  - [ ] AWS Backup이 설정되어 있는가?
  - [ ] Point-in-time Recovery가 활성화되어 있는가?
  - [ ] 백업 주기와 보존 기간이 적절한가?

### 채점 기준
- **100점:** 쿼리 패턴에 최적화된 설계, 백업 설정 완료
- **-10점:** 비효율적 인덱스 설계
- **-20점:** 데이터 정합성 문제 또는 백업 미설정

## 8. E7 — Logging & Monitoring (로깅·모니터링) [5%]

### 평가 항목

#### E7-01: 서버 사이드 로깅
- **체크리스트:**
  - [ ] Lambda 함수에서 주요 작업의 시작/종료를 로깅하는가?
  - [ ] 에러 발생 시 stack trace와 컨텍스트 정보를 포함하는가?
  - [ ] 민감한 정보(토큰, 개인정보)가 로그에 포함되지 않는가?
  - [ ] CloudWatch Logs가 설정되어 있는가?

#### E7-02: 클라이언트 사이드 로깅
- **체크리스트:**
  - [ ] 프로덕션 빌드에서 console.log가 제거되었는가?

## 11. AI Agent 지시사항 (Instructions for AI)

이 문서를 컨텍스트로 받은 AI는 아래를 준수합니다:

1. **선행 조건 확인:** Structural Convergence Rules가 100% PASS인지 먼저 확인한다.
2. **순서 준수:** E1(Security)부터 E7(Logging)까지 순서대로 평가한다. 보안은 항상 최우선이다.
3. **페이지별 평가와 병행:** 카테고리별 횡단 평가와 페이지별 종단 평가를 병행할 수 있다.
4. **점수 근거 제시:** 감점 시 반드시 구체적 파일 경로와 코드 라인을 근거로 제시한다.
5. **수정 제안 구체화:** FAIL 항목에 대해 구체적인 수정 코드 또는 방법을 제안한다.
6. **Critical Issue 즉시 보고:** 보안 취약점, 데이터 손실 위험 등 Critical 이슈는 평가 도중이라도 즉시 사용자에게 보고한다.
7. **카테고리별 승인:** 각 카테고리 평가 완료 후 "다음 카테고리로 진행할까요?"로 확인을 받는다.
8. **종합 보고서:** 모든 카테고리 평가 완료 후 9.4 형식의 종합 보고서를 작성한다.
9. **코드 수정 금지:** 이 프레임워크는 평가 전용이다. 수정은 사용자 승인 후 별도 세션에서 진행한다.
10. **기존 체크리스트 연동:** `페이지별_코드검증_질문리스트.md`가 존재하면 해당 문서의 기준도 함께 적용한다.
