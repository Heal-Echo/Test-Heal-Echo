# 비디오 전송 최적화 전략

> 목적: CloudFront 비디오 전송 비용 절감 (월 비용의 90% 이상 차지)
> 예상 절감: 현실적 시나리오 기준 ~$1,070 → ~$200~350 (60~70% 절감)
> 작성일: 2026-04-03

---

## 현재 상황

| 항목 | 현재 |
|------|------|
| 비디오 포맷 | mp4 (H.264/AAC), MediaConvert로 변환 |
| 화질 | 단일 화질 (업로드 원본 기준) |
| 스트리밍 방식 | mp4 직접 다운로드 (프로그레시브) |
| 프리로딩 | 브라우저 기본값 (자동 다운로드 가능) |
| 캐시 설정 | CloudFront 기본값 |

## 전략 1: 적응형 비트레이트 스트리밍 (ABR) — 절감 50~70%

### 무엇인가?
현재는 비디오 파일 전체를 한 번에 다운로드합니다.
ABR은 비디오를 작은 조각(세그먼트)으로 나누고, 인터넷 속도에 따라 화질을 자동 조절합니다.

- WiFi → HD (1080p) 세그먼트 재생
- LTE → SD (480p) 세그먼트로 자동 전환
- 3G → 최저 화질로 전환

### 구현 방법

#### 1-1. MediaConvert 출력에 HLS 추가

현재 `healecho-infra/lambda/ingest-video:index.js`에서 MediaConvert로 mp4만 생성합니다.
HLS(HTTP Live Streaming) 출력을 추가해야 합니다.

**MediaConvert 출력 설정 (3단계 화질):**

| 화질 | 해상도 | 비트레이트 | 10분 비디오 크기 |
|------|--------|-----------|----------------|
| Low | 480p | 1 Mbps | ~75MB |
| Medium | 720p | 2.5 Mbps | ~188MB |
| High | 1080p | 5 Mbps | ~375MB |

HLS는 6~10초 단위 세그먼트(.ts 파일)로 분할되며, 마스터 플레이리스트(.m3u8)가 화질 목록을 제공합니다.

#### 1-2. 프론트엔드에 HLS 플레이어 추가

```bash
npm install hls.js
```

현재 `<video src="xxx.mp4">` 방식을 HLS 플레이어로 교체:
- hls.js 라이브러리 사용 (iOS Safari는 네이티브 HLS 지원)
- 자동 화질 전환 + 수동 화질 선택 UI 추가

#### 1-3. CloudFront 캐시 정책

HLS 세그먼트는 작은 파일(~1MB)이므로 브라우저 캐시에 들어갑니다.
→ 같은 비디오 재시청 시 실제 캐시 효과를 얻을 수 있습니다.

### 비용 절감 원리
- 모바일 사용자 대부분은 480p~720p로 시청 → 전송량 50~70% 감소
- 사용자가 중간에 이탈해도 이미 다운로드한 세그먼트까지만 과금
- mp4 프로그레시브는 파일 전체를 다운로드하지만, HLS는 시청한 부분만 전송

## 전략 2: 화질 제한 (무료/유료 차등) — 절감 30~50%

### 정책

| 사용자 유형 | 최대 화질 | 예상 비디오 크기 (10분) |
|------------|----------|----------------------|
| 무료 체험 | 720p | ~188MB |
| 유료 사용자 | 1080p | ~375MB |

### 구현 방법
- CloudFront Function 또는 Lambda@Edge에서 구독 상태에 따라 HLS 플레이리스트 필터링
- 무료 체험 사용자에게는 720p 이하 세그먼트만 포함된 플레이리스트 반환

### 비용 절감 원리
- 무료 체험 사용자가 전체 트래픽의 ~80% (2,000명 vs 200명)
- 이 80%의 화질을 720p로 제한하면 전송량 대폭 감소

## 전략 3: 브라우저 캐시 최적화 — 절감 10~20%

### 현재 문제
CloudFront 기본 캐시 헤더가 브라우저에 명시적 캐시 지시를 주지 않아, 모바일 브라우저가 비디오를 캐시하지 않을 수 있습니다.

### 구현 방법

CDK에서 CloudFront 응답 헤더 정책 추가:

```
Cache-Control: public, max-age=604800, immutable
```

- `max-age=604800`: 7일간 브라우저 캐시 유지
- `immutable`: 비디오 파일은 변경되지 않으므로 재검증(revalidation) 요청 방지
- HLS 세그먼트(.ts)에 적용 시 모바일에서도 캐시 효과 증가

### S3 메타데이터 설정
비디오 업로드 시 S3 객체에 `Cache-Control` 헤더 자동 설정:

```
s3.putObject({
  ...
  CacheControl: "public, max-age=604800, immutable"
})
```

## 전략 4: 비디오 코덱 최적화 — 절감 20~30%

### 현재
H.264 코덱 사용 (가장 범용적)

### 개선 옵션

| 코덱 | 장점 | 단점 | 브라우저 지원 |
|------|------|------|-------------|
| H.264 (현재) | 100% 지원 | 파일 크기 큼 | 전부 |
| H.265/HEVC | 50% 크기 감소 | iOS/Safari만 완전 지원 | 부분적 |
| VP9 | 50% 크기 감소 | Chrome/Firefox 지원 | 부분적 |
| AV1 | 50~60% 크기 감소 | 인코딩 느림 | 최신 브라우저 |

### 권장 방안
- **당장**: H.264 유지 (범용성)
- **중기**: MediaConvert에서 H.264 + H.265 이중 출력 → 기기별 자동 선택
- ABR(전략 1)과 결합하면 가장 효과적

## 전략 5: 프리로딩 방지 — 절감 10~15%

### 현재 문제
`<video>` 태그의 기본 `preload` 설정이 브라우저마다 다릅니다.
일부 브라우저는 페이지 로드 시 비디오를 자동으로 다운로드합니다.
→ 사용자가 재생하지 않아도 데이터 소모

### 구현 방법

모든 `<video>` 태그에 적용:

```html
<video preload="none" poster="/thumbnail.webp">
```

- `preload="none"`: 재생 버튼을 눌러야 다운로드 시작
- `poster`: 비디오 대신 썸네일 이미지를 먼저 표시 (수 KB)

### 프론트엔드 수정 대상 파일
- 비디오 재생 관련 모든 컴포넌트에서 `<video>` 태그 확인
- HLS 플레이어 사용 시에도 자동 재생 방지 설정 필요

## 구현 우선순위

| 순위 | 전략 | 절감 효과 | 구현 난이도 | 소요 시간 |
|------|------|----------|-----------|----------|
| **1** | **프리로딩 방지** | 10~15% | 쉬움 | 1일 |
| **2** | **브라우저 캐시 최적화** | 10~20% | 쉬움 | 1일 |
| **3** | **화질 제한 (무료/유료)** | 30~50% | 보통 | 3~5일 |
| **4** | **적응형 비트레이트 (ABR)** | 50~70% | 어려움 | 1~2주 |
| **5** | **코덱 최적화** | 20~30% | 보통 | 3~5일 |

### 권장 실행 순서
1. **즉시 (1~2일)**: 전략 5 (프리로딩 방지) + 전략 3 (캐시 최적화)
   → 코드 변경 최소, 즉시 절감 효과
2. **1주차**: 전략 2 (화질 제한)
   → 무료 체험 사용자 720p 제한으로 큰 절감
3. **2~3주차**: 전략 1 (ABR/HLS)
   → 가장 큰 절감이지만 구현 복잡. MediaConvert 설정 + 프론트엔드 HLS 플레이어 필요
4. **이후**: 전략 4 (코덱 최적화)
   → ABR 구현 후 자연스럽게 적용 가능
