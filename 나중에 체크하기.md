## login
주의가 필요한 점
1. 프롬프트 범위 외 파일 변경 (16개 파일)
커밋에 로그인 리팩토링 외의 변경이 포함되어 있습니다: globals.css(+25줄), HeroProgramsClient.tsx, landing/page.tsx, constants.ts, 그리고 새 API 라우트 auth/exchange/route.ts(+47줄). 이것들은 프롬프트에서 요청하지 않은 변경입니다. 하나의 커밋에 섞여 있어서, 만약 리팩토링만 롤백해야 할 때 다른 변경까지 같이 되돌려야 하는 위험이 있습니다.
2. SignupView의 Props가 많음 (37개)
SignupView에 전달되는 props가 37개나 됩니다. 기능적으로는 문제없지만, 나중에 유지보수할 때 부담이 됩니다. 관련 props를 그룹으로 묶는 객체 패턴(예: passwordState, formState)을 도입하면 더 깔끔해질 수 있습니다. 지금 당장 고칠 필요는 없고, 향후 개선 사항으로 기억해두면 좋습니다.
3. "use client" 지시어 위치
프롬프트에서 "use client"는 page.tsx에만 유지하라고 했는데, 실제로 각 뷰 컴포넌트에 "use client"가 없는 것은 맞습니다. 다만 뷰 컴포넌트들이 useState의 dispatch 함수(예: setShowLoginPw((v) => !v))를 직접 호출하고 있어서, Next.js가 이를 클라이언트 컴포넌트로 올바르게 처리하는지 빌드 테스트에서 확인이 필요합니다. (page.tsx에서 import하므로 자동으로 클라이언트 바운더리에 포함되어 정상 동작할 것입니다.)
4. PwRule 타입 중복 정의
PwRule 인터페이스가 SignupView.tsx와 ForgotStep2View.tsx에 각각 따로 정의되어 있습니다. 공통 타입 파일로 빼면 더 좋습니다.

## home
 향후 별도 작업으로 권장하는 사항:
  - 토큰 저장소를 localStorage → httpOnly 쿠키로 전환
  - API 프록시에서 JWT 서명 검증 추가 여부 검토
  1. 토큰 저장소를 localStorage → httpOnly 쿠키로 전환
지금 상태: 로그인하면 받는 인증 토큰(JWT)을 브라우저의 localStorage에 저장하고 있어요.
문제점: localStorage는 JavaScript로 누구나 읽을 수 있어요. 만약 해커가 악성 스크립트를 페이지에 심으면(XSS 공격), 토큰을 훔쳐갈 수 있어요.
비유하자면: 집 열쇠를 현관 앞 화분 밑에 두는 것과 같아요. 편하지만, 누군가 화분을 들추면 바로 가져갈 수 있죠.
httpOnly 쿠키로 바꾸면: 토큰이 브라우저 쿠키에 저장되는데, httpOnly 설정이 붙으면 JavaScript로는 아예 접근이 불가능해요. 서버와 통신할 때만 자동으로 전송돼요.
비유하자면: 열쇠를 금고에 넣어두는 거예요. 본인이 문을 열 때 자동으로 작동하지만, 다른 사람이 손으로 꺼낼 수는 없어요.
단, 주의할 점: Heal Echo는 나중에 모바일 앱도 연동할 예정이잖아요. 모바일 앱에서는 쿠키 방식이 잘 안 맞을 수 있어서, 웹은 httpOnly 쿠키, 모바일은 기존 방식 이렇게 나눠서 설계해야 해요. 그래서 바로 적용하기보다는 계획을 잘 세운 후에 진행하는 게 좋아요.

2. API 프록시에서 JWT 서명 검증 추가 여부 검토
지금 상태: 프론트엔드의 API 프록시(Next.js의 /api/ 경로)가 요청을 백엔드로 전달할 때, 토큰이 진짜인지 검증하지 않고 그냥 넘겨주고 있어요.
비유하자면: 건물 1층 안내데스크에서 출입증을 보여주면, "이게 진짜 출입증인지"는 확인하지 않고 그냥 엘리베이터로 안내해주는 거예요. 최종 검증은 각 층의 사무실(백엔드)에서 하지만, 1층에서 미리 걸러주면 더 안전하죠.
서명 검증을 추가하면: 프록시 단계에서 "이 토큰이 위조되지 않았는지" 미리 확인해서, 가짜 토큰은 백엔드까지 가기 전에 차단할 수 있어요.
장점은 백엔드 부담이 줄고, 잘못된 요청을 일찍 걸러낼 수 있다는 거예요. 다만 프록시에 Cognito 공개키 설정 등 추가 작업이 필요해요.
